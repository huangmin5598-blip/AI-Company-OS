# AI Company OS - 双轨系统 (Dual-Track System)
**版本**: 1.0
**生效日期**: 2026-03-26

---

## 一、系统结构

```
User Request
    ↓
Task Routing（任务路由）
    ↓
┌─────────────────────────────────┐
│  Project Flow    │  On-demand Flow │
│  (项目流程)      │  (按需流程)     │
└─────────────────────────────────┘
    ↓                    ↓
产出资产              提升效率
```

---

## 二、Task Routing（任务路由 - 必须执行）

### 2.1 判断逻辑

每个请求必须先判断类型：

| 判断维度 | Project Task | On-demand Task |
|----------|--------------|----------------|
| **定义** | 项目型、周期性、长期维护 | 临时性、一次性、验证/调研类 |
| **特征** | 需要多 Agent 协作 | 单次执行即可 |
| **示例** | novel-v1 每日生产 | "帮我查个数据"、"生成一张图" |
| **流向** | TASK-POOL → 多 Agent | On-demand Protocol |

### 2.2 路由决策树

```
收到请求
    ↓
是项目型需求吗？（周期性/多Agent/资产沉淀）
    ↓ YES        ↓ NO
Project Flow    On-demand Flow
    ↓
进入 TASK-POOL   执行 On-demand Protocol
```

## 三、Project Flow（项目流程）

### 3.1 特征
- 使用 TASK-POOL 管理
- 多 Agent 协作链
- 自动推进
- 产出资产沉淀

### 3.2 当前支持项目
| Project ID | 类型 | 周期 | Agent 链 |
|------------|------|------|-----------|
| novel-v1 | 小说生产 | Daily | lead-novel → story-editor → writer → review-editor → export |
| research-agent | 机会研究 | Weekly | lead → research → export |
| hub-v1 | AI 独立站 | NONE | TBD |
| sticker-v1 | 表情包工具 | NONE | TBD |

### 3.3 执行规则
- ✅ 必须写入 TASK-POOL
- ✅ 必须按 Agent 链串行执行
- ✅ 失败立即标记并上报
- ✅ 产出必须沉淀到项目目录

---

## 四、On-demand Flow（按需流程）

### 4.1 执行协议（必须遵守）

**Step 1: 评估风险**
- 评估请求风险等级（低/中/高）
- 评估时间/资源消耗

**Step 2: 提供方案**
- 必须提供 2 种执行方案
- 格式：
  ```
  方案A（稳定优先）：[描述] - 风险低，时间适中
  方案B（效率优先）：[描述] - 风险中，可能更快
  ```

**Step 3: 默认选择**
- 默认选择方案A（稳定优先）
- 用户确认后才执行方案B

**Step 4: 执行与 Fallback**
- 直接执行选定的方案
- 失败立即 fallback 到备用方案
- 仍失败 → 上报用户

### 4.2 示例

**请求**: "帮我用 ComfyUI 生成一张图"

```
方案A（稳定优先）：
- 使用已有 SD1.5 模型
- 优点：模型已在本地，预计 1 分钟内完成
- 缺点：质量可能不如最新版

方案B（质量优先）：
- 需要先下载 Z-Image Turbo 模型（12GB）
- 优点：生成质量更高
- 缺点：下载需要 30 分钟+

默认选择方案A，执行失败后 fallback 到方案B
```

### 4.3 On-demand 任务分类

| 类别 | 示例 | 推荐模型 |
|------|------|----------|
| 信息查询 | 搜索、查资料 | 本地/云端均可 |
| 内容生成 | 写文案、总结 | 云端优先 |
| 工具执行 | 生成图片、执行代码 | 按复杂度选择 |
| 调研分析 | 市场分析、竞品调研 | 云端优先 |

## 五、禁止行为

❌ **不允许直接执行高风险工具**
- 如：删除文件、执行外部命令、发送消息
- 必须先评估 + 提供方案

❌ **不允许卡住不反馈**
- 遇到问题 30 秒内必须响应
- 可以说"需要一点时间查一下"

❌ **不允许死磕单方案**
- 方案失败必须立即 fallback
- 不允许连续失败 2 次以上

❌ **不允许混用项目与临时流程**
- 不能把临时任务写入 TASK-POOL
- 不能把项目任务当作 On-demand 处理

## 七、执行检查清单

每次收到请求时：

```
[ ] 1. 路由判断：是 Project 还是 On-demand？
[ ] 2. 如果是 Project → 进入 TASK-POOL
[ ] 3. 如果是 On-demand → 启动 On-demand Protocol
[ ] 4. 提供 2 个方案（除非明确低风险）
[ ] 5. 默认选择稳定方案
[ ] 6. 执行 → 失败立即 fallback
[ ] 7. 完成 → 产出资产 OR 反馈结果
```

## 九、Task Routing 快速判断清单（终极版）

### 🚀 一句话决策法
这个需求能不能在1步内完成？
- 能 → On-demand
- 不能 → Project

### 🧠 标准判断表

| 判断问题 | YES | NO |
|----------|-----|-----|
| 需要多步骤吗？ | Project | 继续判断 |
| 需要多Agent吗？ | Project | 继续判断 |
| 需要长期跟踪吗？ | Project | 继续判断 |
| 会产出资产吗？ | Project | 继续判断 |
| 只是一次性处理吗？ | On-demand | Project |

### 🔥 最终判断规则
满足任意2条 → Project
全部不满足 → On-demand

### ⚠️ 决策优先级
- 宁可先 On-demand → 再升级
- 不要一上来就 Project

## 十一、升级/降级判断

### On-demand → Project（升级条件）
- 做着做着变复杂
- 需要多步骤
- 需要持续做
- 需要复用结果
→ 立即升级成项目

### Project → On-demand（降级条件）
- 只是小修小改
- 不需要继续跟踪
→ 直接降级处理
