# Build Log: v0.9.2 — External Runtime Connector MVP

## Background

After v0.9's Code-Capable Runtime Bridge (Codex + Claude Code integration), the system had a working bridge but was still tied to specific local agents. The next logical step was clear:

> **Decouple the runtime from the agent.**

The key insight: AI Company OS should not care *which* agent runs a task. It should have a unified Runtime access layer — supporting local agents, experimental agents, and eventually cloud agents — all behind the same protocol.

## Setup / Change

This version introduced 5 changes across the backend:

### 1. External HTTP Agent Adapter Template (`external_http_adapter.py`)

A generic `RuntimeAdapter` implementation for remote agents. Key design decisions:

- **endpoint_url whitelist** — parsed via `urlparse`, only allows `http://localhost` / `http://127.0.0.1` / `https://`; blocks `file://`, `ftp://`, metadata IPs (169.254, 10.x, 172.16-31.x, 192.168.x)
- **Auth via env vars only** — `auth_env` field in config stores the env var name (e.g. `CLOUD_HERMES_TOKEN`); value read from `os.environ.get()` at runtime. No tokens in config, git, or database.
- **execute() hard-disabled** — returns `{"status": "unsupported", "error": "Remote execution disabled in v0.9.2"}` even if path is configured
- **Disabled runtimes** — `enabled=0` causes health_check to skip entirely; zero remote touch

### 2. Claude Code Experimental Integration (`claude_code_stub.py`, `claude_adapter.py`)

Three attempts to get Claude Code working:

| Attempt | Approach | Result |
|:--------|:---------|:-------|
| 1 | `claude -p 'health check'` (v2.1.47) | ❌ Server refused — version too old |
| 2 | `claude --acp --stdio` (ACP mode) | ❌ stdin blocked in non-TTY — `read_line()` hangs |
| 3 | Upgrade to v2.1.150 + `claude -p '...'` | ✅ **25s generate_plan** — works in non-interactive mode |

**Key learning:** Claude Code v2.1.47 was too old. Upgraded via npm to v2.1.150 (brew path conflict resolved). ACP mode is fundamentally incompatible with FastAPI thread-based execution.

### 3. Runtime Registry Expansion (`seed_runtimes.py`)

Added 3 cloud disabled slots:
- `cloud-openclaw` (external, enabled=0)
- `cloud-hermes` (external, enabled=0)
- `minimax-agent` (external, enabled=0)

Total runtime count: **8** (4 enabled + 4 disabled)

### 4. CCR runtime_id Parameter (`code_change_requests.py`)

Added `validate_runtime()` function that:
1. Looks up runtime by ID in the registry
2. Returns 422 if runtime not found
3. Returns 422 if runtime exists but disabled
4. For experimental runtimes (claude-code), blocks patch/apply/rollback with 422

All CCR endpoints accept optional `runtime_id` (defaults to `codex`).

## Key Decisions

1. **execute() hard-disabled** — not just "no-op" but clearly communicates "unsupported" in response body
2. **Auth_env pattern** — storage of env var name rather than value means config can be committed safely
3. **White list over black list** — explicit allow patterns rather than trying to block all dangerous schemes
4. **Claude Code as experimental** — not "incomplete" but "experimental" — intentionally limited capabilities
5. **No frontend changes** — v0.9.2 is purely backend extension; UI for runtime selection deferred

## Risk Mitigation

| Risk | Mitigation |
|:-----|:-----------|
| Cloud runtime misconfiguration | disabled by default; enabled=0 skips all network calls |
| Accidental remote execution | execute() hard-disabled at adapter level before any transport code runs |
| Auth token leak | env var only; no token storage in config, database, or git |
| Unsafe endpoint URLs | urlparse whitelist blocks 5 dangerous patterns |
| Claude Code regression | experimental flag means failure doesn't block other runtimes |

## Known Issues

1. **Claude Code ACP mode blocked** — `claude --acp --stdio` hangs in non-TTY due to stdin readline blocking. Workaround: use `claude -p '...'` for non-interactive commands.
2. **Cloud runtimes are placeholders** — no real cloud agent connected yet. Config only.
3. **No Auto-select** — user must manually specify `runtime_id`. Future: task-based auto-routing.
