### 1. 双层 metadata 载体

Workflow metadata 必须同时保存在两个载体中：

| 阶段 | 载体 | 内容 |
|:-----|:-----|:------|
| **Draft 阶段** | Draft 文件 front matter | workflow_id / step_id / template / depends_on / outputs |
| **Work Order 阶段** | WO metadata_json | 从 Draft front matter 复制 + 补充 asset_id |

**Draft 文件 front matter 示例**：

```markdown
---
workflow_id: WF-20260517-001
workflow_step_id: step_1_review
workflow_template: decision_followup_workflow
step_index: 0
total_steps: 3
depends_on: []
outputs:
  - asset_type: decision_summary
    required: true
    status: pending
---
```

Draft 转 Work Order 时，这些 metadata 字段复制到 WO 的 `metadata_json`。

### 2. Workflow Plan Record（Run Ledger）

| 字段 | 类型 | 说明 |
|:-----|:-----|:------|
| `workflow_id` | string | 自动生成，格式 `WF-YYYYMMDD-NNN` |
| `template` | string | 模板名 |
| `status` | string | active / completed / cancelled |
| `total_steps` | int | 模板 step 数量 |
| `current_step_index` | int | 当前进行到的 step |
| `created_via` | string | 触发来源（cli / api） |
| `context_summary` | string | 用户提供的上下文 |

**Work Order metadata_json 扩展**（每个 WO 增加以下字段）：

```json
{
  "workflow_id": "WF-20260517-001",
  "workflow_step_id": "step_1_review",
  "workflow_template": "decision_followup_workflow",
  "step_index": 0,
  "total_steps": 3,
  "depends_on": [],
  "consumes_asset": null,
  "outputs": [
    {
      "asset_type": "decision_summary",
      "asset_id": null,
      "status": "pending"
    }
  ],
  "workflow_step_status": "pending",
  "blocked_reason": null
}
```

### 2. Template 定义

**路径**: `scripts/templates/`

**Schema**:

```yaml
workflow:
  id: decision_followup_workflow
  name: "Decision Follow-up Workflow"
  description: "从 Founder Decision 到可执行 follow-up 任务"
  version: "0.1"

steps:
  - step_id: decision_review
    task_type: decision_review
    action: create_draft
    action_class: read_only
    outputs:
      - asset_type: decision_summary
        required: true
        description: "决策上下文摘要，供下一步 draft 使用"

  - step_id: draft_next_action
    task_type: draft_generation
    action: draft_from_decision
    action_class: safe_output
    depends_on:
      - step: decision_review
        consumes_asset: decision_summary
    outputs:
      - asset_type: followup_draft
        required: true
        description: "下一步的 Draft，可转为 Work Order"

  - step_id: execute_work_order
    task_type: work_order_execution
    action: create_work_order
    action_class: approval_required
    depends_on:
      - step: draft_next_action
        consumes_asset: followup_draft
    outputs:
      - asset_type: execution_result
        required: false
        description: "执行结果"
```

**字段说明**:

| 字段 | 必须 | 说明 |
|:-----|:----:|:------|
| `step_id` | ✅ | 唯一 step ID |
| `task_type` | ✅ | 任务类型分类 |
| `action` | ✅ | 对应 capability-boundary 的 action |
| `action_class` | ✅ | 对应 boundary 分类（决定 policy check） |
| `outputs` | ✅ | 本 step 产出物声明（含 required 标志） |
| `depends_on` | ❌ | 前置依赖（第一项为空） |
| `consumes_asset` | ❌ | 消费的 asset type（在 depends_on 内） |
| `completion_criteria` | ✅ | step 完成判断标准 |

**completion_criteria 可选值**：

| 类型 | 适用 action_class | 完成条件 |
|:-----|:-----------------|:---------|
| `asset_created` | read_only / safe_output | 产出资产已登记到 Asset Registry |
| `work_order_completed` | approval_required | Work Order 状态为 completed |
| `result_synced` | approval_required | Work Order completed + 结果已回写 |

### 3. Capability Boundary 新增 action

```yaml
actions:
  # ── Workflow ──
  create_workflow:
    class: safe_output
    description: "创建新的 workflow"
  run_workflow_next:
    class: approval_required
    description: "前进 workflow 到下一步"
  workflow_blocked_resolve:
    class: read_only
    description: "手动解除阻塞"
  workflow_skip_step:
    class: approval_required
    description: "跳过 workflow 某一步"
  workflow_cancel:
    class: approval_required
    description: "取消整个 workflow"
```

### 文件清单

| 1 | 新建 | `scripts/workflow_runner.py` | 主 CLI |
| 2 | 新建 | `scripts/workflow_helpers.py` | 工具函数（ID 生成、Run Ledger 写入、Asset 登记） |

### 命令实现

#### `create`

```
python3 scripts/workflow_runner.py create --template decision_followup_workflow [--context "..."]

行为:
1. 读取 scripts/templates/<template>.yaml
2. 校验 template schema（steps 数组 > 0，outputs 声明等）
3. 生成 workflow_id (WF-YYYYMMDD-NNN)
4. 写入 Run Ledger: workflow_created
5. 登记 Asset Registry: workflow_plan
6. 调用 ceo_cmd 生成 Step 1 的 Draft（走 policy_resolver）
7. 输出: Workflow created + Step 1 Draft ready for Founder
```

#### `status`

```
python3 scripts/workflow_runner.py status <workflow_id>

行为:
1. 从 Run Ledger 读取 workflow 记录
2. 扫描各 step 的 WO 状态
3. 输出表格: step_id / status / wo_id / blocked_reason / output_assets
4. 标注当前进度 (2/3 completed)
```

#### `next`

```
python3 scripts/workflow_runner.py next <workflow_id>

行为:
1. 读取 workflow plan + 模板
2. 找到当前已完成的最末 step（按 completion_criteria 判断）
3. 检查下一个 step 的 depends_on 是否全部 completed
   └── 否 → 输出 BLOCKED + blocked_reason → 写入 workflow_blocked 事件
   └── 是 → 继续
4. 读取前置 step 的 output asset（三层 fallback）
   优先级 1: step metadata.outputs[].asset_id
   优先级 2: Asset Registry 按 source_work_order / asset_type 查询
   优先级 3: WO result_summary / draft metadata 生成 fallback context
   均找不到 → BLOCKED, blocked_reason=missing_required_asset
5. 生成 Draft context（含前置 asset 的 summary）
6. 调用 ceo_cmd 生成下一步 Draft（走 policy_resolver，Draft front matter 写入 workflow metadata）
7. 写入 Run Ledger: workflow_step_created + workflow_step_unlocked
8. 输出: Step N Draft ready for Founder review
```

#### `resolve / skip / cancel`

```
resolve <workflow_id> <step_id>
  → 将 step 状态从 blocked 改为 pending
  → 写入 Run Ledger: workflow_block_resolved
  → 不自动触发 next

skip <workflow_id> <step_id>
  → 标记 step skipped + skip_reason
  → 写入 Run Ledger: workflow_step_skipped
  → 生成 skip_context asset（替代原计划资产）
    {
      "asset_type": "workflow_step_context",
      "summary": "Step <step_id> was skipped by Founder. The expected <consumes_asset> is unavailable.",
      "status": "skipped_context"
    }
  → 后续 step 的 consumes_asset 变为 skip_context（不静默置 null）
  → 不自动触发 next

cancel <workflow_id>
  → 标记所有未完成 step cancelled
  → 更新 workflow status → cancelled
  → 写入 Run Ledger: workflow_cancelled
```

## 风险 & 缓解

| 风险 | 概率 | 影响 | 缓解 |
|:-----|:----:|:----:|:-----|
| Draft context 注入质量差，next 生成的 Draft 需要大量人工修改 | 中 | 中 | context 只包含 asset summary + metadata，不包含全文；Founder review 机制兜底 |
| 嵌套 workflow（一个 WO 本身是另一个 workflow 的 step）被自然提出 | 高 | 中 | v0.30 不做嵌套，明确写在设计文档中。如果有人问，推给 v0.31 |
| 用户期待 next 自动执行而非生成 Draft | 中 | 高 | 在 Founders' Handbook 中明确写清楚：every step requires Founder approval |
| Step 数量少（2-3），用户觉得"还不如手动创建 WO" | 中 | 低 | 说明这是 v0.30 的起点，后续版本会增加并行/条件分支。第一版验证机制 |
| skip 后依赖该 step 的后续 step 缺少 asset 输入 | 中 | 低 | skip 时生成 skip_context asset 替代原计划资产，后续 Draft 明确提示"上一步已被跳过" |
