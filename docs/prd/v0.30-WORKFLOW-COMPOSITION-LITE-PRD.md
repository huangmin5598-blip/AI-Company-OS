## 三、Workflow Schema

### 双层 metadata 载体

Workflow metadata 必须同时保存在两个载体中：

| 阶段 | 载体 | 内容 |
|:-----|:-----|:------|
| **Draft 阶段** | Draft 文件 front matter | workflow_id / step_id / template / depends_on / outputs |
| **Work Order 阶段** | WO metadata_json | 从 Draft front matter 复制 + 补充 asset_id |

**Draft 文件结构示例**：

```markdown
---
workflow_id: WF-20260517-001
workflow_step_id: step_1_review
workflow_template: decision_followup_workflow
step_index: 0
total_steps: 3
depends_on: []
outputs:
  - asset_type: decision_summary
    required: true
    status: pending
---
... Draft 正文 ...
```

Draft 转 Work Order 时，将这些 metadata 字段复制到 WO 的 `metadata_json`，并补充实际产出资产的 `asset_id`。

### 字段设计

使用 `metadata_json` 存储 workflow 信息，**不修改 Work Order 表结构**。

每个 Work Order 的 `metadata_json` 扩展：

```json
{
  "workflow_id": "WF-20260517-001",
  "workflow_step_id": "step_1_review",
  "workflow_template": "decision_followup_workflow",
  "depends_on": [],
  "consumes_asset": null,
  "outputs": [
    {
      "asset_type": "decision_summary",
      "asset_id": null,
      "status": "pending"
    }
  ],
  "workflow_status": "active",
  "blocked_reason": null,
  "step_index": 0,
  "total_steps": 3
}
```

每个 workflow 自身也有一条 Run Ledger 记录（`workflow_created` 事件），包含 workload-level metadata。

### Workflow Step 状态机

```
     created
       │
    in_progress
       │
  ┌────┴────┐
  │         │
completed  blocked
  │         │
  │     ┌───┴───┐
  │     │       │
  │  resolved  skipped
  │     │       │
  └─────┴───────┘
       │
  (next unlocked)
```

## 四、模板设计

### Step Completion Criteria

每个 step 必须声明 `completion_criteria`，否则 `workflow_runner.py next` 不知道何时解锁下一步。

| 类型 | 适用 action_class | 完成条件 |
|:-----|:-----------------|:---------|
| `asset_created` | read_only / safe_output | 产出资产已登记到 Asset Registry |
| `work_order_completed` | approval_required | Work Order 状态为 completed |
| `result_synced` | approval_required | Work Order completed + 结果已回写 |

**模板示例**：

```yaml
- step_id: decision_review
  action_class: read_only
  outputs:
    - asset_type: decision_summary
      required: true
  completion_criteria:
    type: asset_created
    required_asset_type: decision_summary

- step_id: execute_work_order
  action_class: approval_required
  completion_criteria:
    type: result_synced
    required_asset_type: execution_result
```

### 模板 1 — Decision Follow-up Workflow

**用途**：Founder / CEO Brief 产生一个 decision 后，转成可执行 follow-up 任务。

```
Founder Decision
  ↓
step_1: decision_review (read_only)
  → 确认决策上下文
  → output: decision_summary (asset)
  ↓
step_2: draft_next_action (safe_output)
  → 消费 decision_summary
  → 生成 follow-up Draft
  → output: followup_draft (asset)
  ↓
step_3: execute_work_order (approval_required)
  → 消费 followup_draft
  → 创建 Work Order (走 policy_resolver)
  → Founder approve-dispatch
  → 执行
  → result_synced
  → output: execution_result (asset)
  ↓
workflow_completed
```

### 模板 2 — Opportunity Follow-up Workflow

**用途**：已有 opportunity_signal / opportunity_card（或 Founder 确认的方向），做深度研究 + 验证。

```
Opportunity Signal / Founder-Approved Direction
  ↓
step_1: deep_research (safe_output)
  → 读 AI-Knowledge-OS 已有知识
  → 产出 research_brief (asset)
  ↓
step_2: validation_plan (safe_output)
  → 消费 research_brief
  → 产出验证计划/机会卡片 (asset)
  ↓
step_3: execute_validation (approval_required)
  → 消费 validation_plan
  → 创建 Work Order (走 policy_resolver)
  → Founder approve-dispatch
  → 执行
  → result_synced
  ↓
workflow_completed
```

**注意**：此模板的**起点**不是"文章抓取"，而是**已有机会信号**（来自 Founder 判断、AI 识别、知识库分析等）。文章→知识的摄取由 Hermes + AI-Knowledge-OS 负责，不在此模板范围内。

## 五、Workflow Runner CLI

### 命令设计

```bash
# 创建 workflow（生成 workflow record + step 1 的 Draft）
python3 scripts/workflow_runner.py create --template decision_followup_workflow
python3 scripts/workflow_runner.py create --template decision_followup_workflow --context "跟进 XX 方向的调研决策"

# 查看状态
python3 scripts/workflow_runner.py status <workflow_id>

# 前进到下一步（检查依赖→读前置 asset→生成下一步 Draft）
python3 scripts/workflow_runner.py next <workflow_id>

# 阻塞处理
python3 scripts/workflow_runner.py resolve <workflow_id> <step_id>
python3 scripts/workflow_runner.py skip <workflow_id> <step_id>
python3 scripts/workflow_runner.py cancel <workflow_id>
```

### `next` 命令职责

`next` 不是简单的 "创建下一个 WO"，而是：

```
1. 检查前置 step 是否 completed（按 completion_criteria）
2. 找到前置 step 的 output asset（三层 fallback）
   优先级 1: step metadata.outputs[].asset_id
   优先级 2: Asset Registry 按 source_work_order / asset_type 查询
   优先级 3: WO result_summary / draft metadata 生成 fallback context
   均找不到 → BLOCKED, blocked_reason=missing_required_asset
3. 读取 asset summary / path / metadata
4. 注入下一步 Draft context
5. 生成下一个 Work Order Draft（走 policy_resolver，Draft front matter 含 workflow metadata）
6. 记录 workflow_step_created 事件到 Run Ledger
7. 等 Founder 审批 → create WO → approve-dispatch
```

**安全边界**：
- `next` 不自动 approve
- `next` 不自动 execute
- `next` 不跳过 Founder 审批
- `next` 在 step 未完成时生成 BLOCKED 状态

### Blocked 处理

| 命令 | 行为 | Run Ledger 事件 |
|:-----|:-----|:----------------|
| `resolve` | Founder 确认阻塞已解除，step 恢复 in_progress | `workflow_block_resolved` |
| `skip` | 跳过该 step，生成 `skipped_context` asset（替代原计划资产），不静默置 null | `workflow_step_skipped` |
| `cancel` | 取消整个 workflow，所有未完成 step 标记 cancelled | `workflow_cancelled` |

**skip_context 说明**：skip 后不是把后续 step 的 `consumes_asset` 置 null，而是显式生成一个 `workflow_step_context` 资产：

```json
{
  "asset_type": "workflow_step_context",
  "summary": "Step validation_plan was skipped by Founder. The expected opportunity_card asset is unavailable.",
  "status": "skipped_context"
}
```

后续 step 消费此资产时，Draft 将明确提示："上一步已被 Founder 跳过，因此本步骤缺少原计划输入。"

## 六、Run Ledger / Asset Registry / Policy 接入

### Run Ledger 事件

| 事件类型 | 触发点 |
|:---------|:-------|
| `workflow_created` | create 命令 |
| `workflow_step_created` | next 命令生成 Draft |
| `workflow_step_completed` | WO 执行完成 |
| `workflow_step_unlocked` | 前置 step completed → 解锁 |
| `workflow_blocked` | next 检测到依赖未满足 |
| `workflow_block_resolved` | resolve 命令 |
| `workflow_step_skipped` | skip 命令 |
| `workflow_cancelled` | cancel 命令 |
| `workflow_completed` | 最后一步 completed |

### Asset 类型

| asset_type | 说明 |
|:-----------|:-----|
| `workflow_plan` | workflow create 时生成的计划记录 |
| `workflow_step_context` | next 注入的 Draft context |
| `workflow_step_output` | step 执行后的产出物 |

### Policy Resolver

Policy Resolver 必须在以下入口保持生效：

- `workflow create` — 检查 actor 是否有权限创建 workflow
- `workflow next` — 检查下一步动作是否允许
- Draft generation — 走 ceo_cmd 现有 policy check
- approve-dispatch — 走 work_order_control 现有 policy check

## 八、验收标准

| # | 标准 | 验证方式 |
|:-:|:-----|:---------|
| 1 | workflow 模板 YAML 可解析 | `python3 scripts/workflow_runner.py create --template ...` |
| 2 | 至少 2 个模板完成 | `scripts/templates/` 目录下有 2 个 YAML 文件 |
| 3 | create 能创建 workflow record + step 1 Draft | Run Ledger 有 `workflow_created` 事件 |
| 4 | step metadata 包含 workflow_id / step_id / outputs / depends_on | 查看 WO metadata_json |
| 5 | 前置 step 未完成时，next 返回 BLOCKED | `workflow_blocked` 事件 |
| 6 | 前置 step completed 后，next 可生成下一步 Draft | `workflow_step_created` 事件 |
| 7 | resolve 能解除阻塞，skip 能跳过，cancel 能取消 | 对应 Run Ledger 事件 + status 变更 |
| 8 | workflow events 写入 Run Ledger | `SELECT * FROM run_ledger WHERE event_type LIKE 'workflow_%'` |
| 9 | workflow assets 写入 Asset Registry | `register_asset(type='workflow_plan')` |
| 10 | policy_resolver 在 step 创建 / approve-dispatch 时生效 | 已有 policy check 不破坏 |
| 11 | 不自动 approve / execute | 每次 step 生成 Draft 后需 Founder 审批 |
| 12 | 现有测试不破坏 | `python3 -m pytest tests/ -v` |
