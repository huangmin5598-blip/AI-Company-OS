# Monitor Framework Lite — Architecture Preview

> **v0.5 target designed in v0.4.1 to ensure RuntimeAdapter Protocol compatibility.**
> This is a forward-looking architecture document — implementation begins in v0.5.

---

## 1. Why Monitor Framework

The Company Loop (v0.2) is reactive — it responds to alerts and commands. The CEO Agent (v0.3) is directive — it acts on founder intent. But the system has no self-observation capability: it doesn't know if agents are stuck, memory is stale, costs are spiking, or patterns are emerging.

**Monitor Framework Lite** adds the first layer of self-observation. Not full self-improvement (v0.7) — just detection and alerting.

---

## 2. Architecture: Probes → Analyzers → Outputs

```
┌─────────────────────────────────────────────────────────┐
│                     Monitor Framework                      │
│                                                           │
│  ┌──────────┐    ┌────────────┐    ┌──────────────────┐  │
│  │  Probes   │───▶│ Analyzers  │───▶│    Outputs       │  │
│  │ (data)    │    │ (detect)   │    │ (notify/store)   │  │
│  └──────────┘    └────────────┘    └──────────────────┘  │
│       │                │                   │              │
│       ▼                ▼                   ▼              │
│  RuntimeAdapter    Rule Engine         Alert System       │
│  Protocol calls    (YAML config)      → Company Loop      │
└─────────────────────────────────────────────────────────┘
```

### 2.1 Probes (Data Collection)

Probes collect raw data from various sources. Each probe implements a simple interface — collect data, return structured output.

| Probe | Source | Data Collected |
|:------|:-------|:---------------|
| `hermes_probe` | Hermes Agent (via RuntimeAdapter) | Session status, active skills, error rate |
| `cost_probe` | Cost Adapter | Token usage, cost per runtime, trends |
| `memory_probe` | OrgMemory model | Stale entries, low-recall patterns |
| `task_probe` | TaskPool | Stuck tasks, aging approvals |

**Probe interface (planned v0.5):**

```python
class MonitorProbe(Protocol):
    async def collect(self) -> list[MonitorDataPoint]:
        """Collect data points from the source."""
        ...
```

### 2.2 Analyzers (Detection)

Analyzers examine probe data against rules and thresholds.

| Analyzer | What It Detects |
|:---------|:----------------|
| `stuck_task_analyzer` | Tasks in progress > threshold without updates |
| `cost_spike_analyzer` | Cost increase > X% over rolling window |
| `memory_stale_analyzer` | Memory entries not recalled in > N days |
| `error_rate_analyzer` | Runtime error rate > threshold |

**Analyzers are configurable via `config/monitor-rules.yaml`:**

```yaml
analyzers:
  stuck_task:
    enabled: true
    max_age_minutes: 120
    check_interval: 30

  cost_spike:
    enabled: true
    spike_threshold_percent: 50
    window_hours: 24
```

### 2.3 Outputs (Action)

When an analyzer detects something, the output layer decides what to do.

| Output | Action |
|:-------|:-------|
| `alert_output` | Create an Alert → Company Loop |
| `log_output` | Write to analysis records (non-urgent) |
| `memory_output` | Create a Learning Candidate for self-improvement |

**Default flow:**

```
Analyzer detects issue
  → Alert created
    → Alert → Task → Context Pack → Approval → Execute → Review
      → Learning Candidate (if applicable)
```

## 3. RuntimeAdapter Protocol Integration

Monitor Framework relies on RuntimeAdapter Protocol for its probes. This means:

- Probes don't care which runtime is being monitored
- New runtimes automatically get basic monitoring
- The protocol's `health_check()` and `get_cost()` methods feed directly into probes

```
RuntimeAdapter.health_check()    → hermes_probe status data
RuntimeAdapter.get_cost()        → cost_probe token usage data
RuntimeAdapter.get_capabilities() → capability tracking
```

## 4. What v0.5 Delivers

| Feature | Scope |
|:--------|:------|
| **2 Probes** | `hermes_probe` + `cost_probe` (first versions) |
| **3 Analyzers** | stuck_task, cost_spike, error_rate |
| **1 Output** | alert_output → Company Loop |
| **Config** | `config/monitor-rules.yaml` |
| **No UI** | Monitored through existing Alert/Task UI |

### What v0.5 Does NOT Deliver

- Automatic remediation (v0.7)
- Self-improvement loop (v0.7)
- Full dashboard (future)
- Multi-runtime probe orchestration (v0.6+)
