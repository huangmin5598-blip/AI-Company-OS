前面几章其实一直在回答“为什么这样设计”。
而从这里开始，我们要进一步回答：
那到底怎么落地，才不会一上来就把系统搞成一个自己也维护不住的怪物？
所以第 25 章非常重要。
它会把前面所有关于角色、工作流、升级流、共享中心、模型分层的思路，压缩成一个非常现实的落地原则：
第 25 章｜最小可跑版本：前台少，后台多
这是我认为整个系统落地时，最值得优先守住的一条原则之一。
因为很多 AI 公司架构一开始不是死在“想不明白”，
而是死在：
一上来就开太多口子。
也就是：


前台入口太多


可直接对话的 Agent 太多


曝露给自己的角色太多


渠道机器人太多


自动化接线太早、太散、太碎


看起来像“组织很完整”，
实际上很容易变成：


你自己每天被很多 Agent 打扰


项目主管被绕过


执行层口径乱飞


后台角色逐渐失控


系统越来越像一个多人吵架群


所以这一章要讲清楚一个非常重要、而且特别反人性的落地原则：
最小可跑版本，不是把所有角色都搭出来，
而是只先打开那些真正必要的前台入口。
一句话概括就是：
前台少，后台多。

25.1 为什么很多系统一开始就死在“太想一步到位”
这个冲动特别正常。
因为当你已经把架构想得比较完整时，你会天然想：


CEO Agent 有了


项目主管有了


研发中心有了


增长中心有了


客服底座有了


知识平台有了


各项目也都能拆


那是不是干脆一次全配上？
从纸面上看，这种“一次完整搭建”特别诱人。
因为它给人一种很强的完成感：

我已经把公司搭好了。

问题在于，架构图上的完整，和真实世界里的可运行，不是一回事。
一个系统最危险的时候，往往不是它角色不够多，
而是它太早把太多角色变成了活跃入口。
因为一旦入口开多了，会立刻引发这些问题：


谁都能跟你说话


谁都能来汇报


谁都可能越层触发你注意力


谁都可能在还没稳定前就暴露自己的不成熟


你自己很难判断，到底应该用谁来接哪种事


最后你不是拥有了一家公司，
而是拥有了一堆同时朝你发消息的半成品员工。
这就是为什么我一直不建议一开始就追求“全量开放”。
落地系统和设计系统是两回事。
设计时你可以想完整，
落地时你必须克制。

25.2 什么叫“最小可跑版本”
最小可跑版本，不是最小角色数量。
它更准确地说，是：
在不牺牲核心组织结构的前提下，
只开放最少的入口、最少的交互面、最少的运行路径，
先让系统形成真实闭环。
这句话有三个重点。
第一，不牺牲核心组织结构
也就是说，不是为了简化就退回成“一个万能 Agent 先顶着”。
组织原则仍然要在。
第二，只开放最少入口
你不是把所有东西都暴露出来。
你只是先打开最关键的管理入口。
第三，先形成真实闭环
不是把目录和配置写满，
而是真正让一条业务链能跑起来。
所以“最小可跑版本”不是指小得随便，
而是指：
最小，但仍然是一个真正的组织。
这点非常关键。
因为很多人一提 MVP，就容易把东西做成“凑合能跑”。
而这里讲的最小可跑版本，不是凑合，
而是：
用最小暴露面，保住正确结构。

25.3 为什么前台入口越少，系统越容易稳
前台入口，说白了就是：
你平时能直接触达、直接对话、直接管理的 Agent 入口。
这类入口一旦多起来，
你马上就会感受到一种熟悉的混乱：


今天这个 Agent 来给你一个建议


明天那个 Agent 来问你一个问题


后天又一个 Agent 贴给你一堆原始信息


很快你开始分不清：这是谁该说的话、谁不该说的话


这就是为什么前台入口越多，
你越容易重新掉回“人肉协调器”的角色。
而前台入口少，会带来几个非常重要的好处：
第一，你的注意力更集中
你不会被太多低层角色直接打断。
第二，组织层级更容易守住
CEO Agent 和项目主管不会轻易被架空。
第三，系统更容易形成统一口径
项目内问题先在项目内消化，
不是谁都能直接把局部信息甩到你面前。
第四，更容易排错
当系统出问题时，你至少知道是从哪些入口流出来的。
所以从落地角度看，前台入口少不是“保守”，
而是：
给组织留出形成秩序的空间。

25.4 为什么后台角色反而应该比前台更多
这句话第一次听会有点反直觉：
前台少，后台多。
为什么后台反而要多？
因为后台角色才是真正承担具体能力的地方。
比如：


Builder


QA


Research


Content Draft


Content Review


Comment Analyzer


Knowledge Curator


Story Critic


Fusion Agent


Opportunity Scout


Release Checker


这些角色很多时候并不适合直接成为你每天对话的入口。
但它们非常适合存在于系统后台，作为：


被项目主管调用的能力单元


被共享中心调度的专业角色


被工作流触发的后台节点


也就是说，后台角色多不是问题。
问题只是你不要把它们全变成“日常前台入口”。
一家公司后台有很多人很正常。
不正常的是，老板每天同时和所有后台角色高频直聊。
AI 公司也是一样。
所以“后台多”其实意味着：
你允许系统内部有更细的能力拆分，
但不允许这些拆分直接打断你的管理界面。
这才是成熟落地。

25.5 最小可跑版本里，前台最适合先开哪些角色
这个问题要讲得非常落地。
如果你真的在搭第一版系统，我最建议优先打开的前台入口，通常只有这些：
第一：CEO Agent
这是你最重要的总入口。
你用它看全局、看项目状态、看风险、看资源冲突、看需要你拍板的事。
第二：各项目主管 Agent
这是你深入单项目时最合理的入口。
不是直接找执行层，而是通过项目主管了解项目。
第三：极少数真正需要你本人强参与的专项入口
比如某些高价值创意项目、某些必须你亲自判断的关键环节。
但这类入口必须克制。
你会发现，这其实和前面第 10 章讲的指挥流完全一致。
也就是说，落地时最小可跑版本的前台设计，
其实是在用配置方式，把你的管理原则真正固化下来。
不建议一开始直接开放的
通常包括：


各类执行层 Agent


各类共享中心底层角色


客服底座前台


大量自动研究前台


各种细粒度专项机器人


不是因为它们没价值，
而是因为它们太容易打断系统形成秩序。

25.6 为什么“少入口”比“少角色”更重要
这点特别值得单独讲一下。
很多人一说最小版本，就会自然想成：

那我是不是角色也要尽量少？

不一定。
角色是否少，取决于你后台是否需要更细分的能力。
但入口是否少，几乎一定要优先控制。
这两者差别非常大。
少角色
意味着你连后台能力都没准备好。
很多时候这会限制系统真实工作能力。
少入口
意味着你只是控制谁能直接打到管理界面。
后台仍然可以很丰富。
所以一个成熟的最小可跑版本，
很可能是：


后台已经有一定丰富度


但前台看上去非常克制


这反而是健康状态。
因为真正需要先被验证的，不是“后台能力能不能写满”，
而是：
管理界面能不能不被淹没，
核心链路能不能不靠你亲自补位。
只要这点成立，后台能力以后都还能加。
如果这点不成立，你后台加再多也只是加噪音。

25.7 为什么很多自动化都应该等系统先跑顺再接
这是一个特别现实的提醒。
很多人一看到 AI 公司架构，很容易马上想到：


这个能不能自动推送


那个能不能自动触发


这个能不能自动分发


那个能不能自动发帖


这个能不能自动评论


那个能不能自动抓数据再回填


这些当然都可能有价值。
但我非常建议你分清一个先后顺序：
先让组织流转顺，
再接更深的自动化。
为什么？
因为自动化接得越早，
你越可能把尚未稳定的流程“机械放大”。
比如：


还没想清楚谁该汇报谁，就先自动推送一堆信息


还没想清楚口径怎么统一，就先自动对外发很多内容


还没想清楚升级规则，就先自动转发一堆异常


还没想清楚客服边界，就先自动回复用户


这些都特别危险。
因为自动化最大的力量，不是“更聪明”，
而是“更快地重复”。
而如果你让一个不成熟的流程更快地重复，
本质上是在放大错误。
所以最小可跑版本里，我更建议：


自动化少一点


人控多一点


流程先跑顺


关键路径先摸清


真正稳定后，再逐步自动化


这会让你少掉很多后期大修的成本。

25.8 最小可跑版本最重要的，不是“看起来像公司”，而是“真的能闭环”
很多人很容易被架构图迷住。
因为架构图特别容易给人一种完成感。
有 CEO。
有项目主管。
有共享中心。
有后台角色。
有模型路由。
有知识库。
有自动化。
看起来很像一家公司。
问题在于：
看起来像，不代表真的能跑。
最小可跑版本最重要的，不是“像不像完整公司”，
而是：
它有没有至少一条真实业务闭环能跑通。
比如：


一个项目主管能不能真的接目标、调共享能力、收结果、向上汇报


一个内容项目能不能真的跑完“选题—生成—发布—反馈回流”


一个服务项目能不能真的跑完“内容—留资—体验—反馈”


一个运营项目能不能真的跑完“研究—素材—页面—复盘”


只要有一条真实闭环跑通，
你的系统就从“概念设计”进入“真实组织”了。
反过来，如果角色再多、目录再全、配置再漂亮，
但没有一条真正闭环，
那你手里拥有的更多只是：
一套很完整的构想。
而不是一家公司。

25.9 一个成熟的最小可跑版本，最终应该长什么样
如果把这一章收束成一个完整画面，一个成熟的最小可跑版本，通常应该具备这些特征：
第一，前台入口极少
通常只有：


CEO Agent


项目主管 Agent


极少数高价值专项入口


第二，后台能力已经开始成型
不是空架子，
而是真有一些能被调用的角色和共享中心能力。
第三，至少有一条真实业务链能跑通
不是纸上流程，而是真正在运转。
第四，信息和升级流已经有初步边界
不是谁都能来找你，也不是所有问题都靠你人肉判断。
第五，自动化还保持克制
重点先在“能跑顺”，而不是“看起来最智能”。
到这一步，系统虽然还远不是最终形态，
但它已经足够开始承载真实工作、真实反馈、真实迭代。
这就是最小可跑版本真正的意义。
它不是终局。
但它必须是一个正确的起点。

本章小结
最小可跑版本的关键，不是把所有角色都搭出来，
而是：
在保留正确组织结构的前提下，只开放最少的前台入口，先让系统真正跑出闭环。
一套成熟的最小可跑版本，应该遵循：


前台少，后台多


少入口，不一定少角色


先跑顺核心链路，再加更深自动化


先形成真实闭环，再追求全量完整


管理界面尽量收窄，避免创始人重新变成人肉协调器


你真正要追求的，不是“一口气把公司搭满”，
而是：
先让一条对的组织路径真的跑起来。
接下来，我们会进入一个特别落地、也特别关键的部分：
SOUL.md。
因为如果说第 25 章解决的是“前台和后台怎么开”，
那下一章要解决的就是：
这些 Agent 的人格、边界和行为规则到底怎么写，系统才不会漂。

25.4 最小闭环不是先搭满 Agent，而是先定三份执行协议

很多人第一次设计多 Agent 系统时，会犯一个非常典型的错误：

先把所有 Agent 想出来。

例如：

- CEO Agent
- 项目主管 Agent
- 研发 Agent
- 测试 Agent
- 内容 Agent
- 增长 Agent
- 客服 Agent

然后很快就会得到一个看起来非常宏大的架构图。

但在真实实践中我发现：
架构图画出来，并不等于系统可以运行。
真正决定系统是否能跑起来的，其实不是 Agent 数量，而是：
执行协议。

在我们的实践中，一个 AI 公司系统能否真正开始运转，至少要先确定三份协议：

第一：任务卡协议（Task Card）
任何正式任务，都必须通过标准任务卡发出。
任务卡至少需要包含：

- 项目ID
- 当前目标
- 任务类型
- 交付物
- 验收标准
- 边界条件
- 优先级
- 截止时间
- 发起人
- 升级条件

任务卡的作用只有一个：
把模糊需求变成可执行任务。

没有任务卡，Agent 协作很快就会变成：

- 对话
- 临时讨论
- 不可追踪的请求

最终系统会重新退化为“聊天工具”。

第二：任务状态流转协议
任务不是发出去就结束了。
它必须在一个明确的状态系统中流动。

一个最小可用的任务状态系统通常包括：

- 待创建
- 待派发
- 已派发
- 执行中
- 待验收
- 待修改
- 已完成
- 已阻塞
- 已升级

其中最重要的一点是：
验收失败必须能回流。

否则系统只会产生两种任务：

- 完成
- 永远卡住

第三：共享中心接单协议
在多项目环境下，最容易失控的一件事是：
所有人都在找所有人。

如果项目主管可以直接找：

- builder
- qa
- creative
- growth

系统很快会变成一个混乱的 Slack 群。

因此必须规定：
正式接单口只有一个。

项目主管 → 共享中心
共享中心 → 内部分发执行 Agent

这样做有三个好处：

1. 任务入口统一
2. 优先级可以集中管理
3. 执行 Agent 不被多项目同时打扰

一个重要结论
很多人会问：

一个 AI 公司需要多少 Agent 才能跑起来？

我的答案是：

不是 Agent 数量决定系统能否运行，而是任务流转机制。

只要这三份协议存在：

- 任务卡
- 状态流转
- 接单规则

即使只有 4 个 Agent，系统也可以开始运行。
