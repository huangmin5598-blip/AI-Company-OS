如果说前面第 25 章解决的是：


前台怎么收口


后台怎么铺开


最小可跑版本怎么启动


那这一章要解决的，就是另一个会直接影响系统稳定性的核心问题：
第 26 章｜SOUL.md 怎么写，Agent 才不会人格漂移
很多人第一次搭 Agent 系统时，最容易做的一件事，就是给 Agent 写一个 prompt。
比如：


你是一个很专业的客服


你是一个优秀的项目经理


你是一个资深增长专家


你是一个会写代码的工程师


这些当然比什么都不写强。
但只要你真的把系统跑起来，你很快就会发现：
光有一句角色设定，远远不够。
因为 Agent 最大的问题，很多时候不是“它不会干活”，
而是：


它一会儿像这个岗位，一会儿又像那个岗位


它今天很克制，明天就开始越权发挥


它本来该做执行，结果突然开始替你做决策


它本来该保持语气稳定，结果越聊越飘


它本来只负责项目内问题，结果开始给出公司级建议


这就是我说的：
人格漂移。
这里的“人格”，不是文学意义上的人物性格，
而是一个 Agent 在系统里的：


角色边界


行为风格


责任范围


输入输出方式


升级条件


禁止事项


如果这些东西不清楚，
Agent 就很容易在长期运行中慢慢变形。
所以我越来越觉得，Agent 稳不稳定，靠的不只是模型强不强，
更靠：
它有没有一份写得足够清楚的 SOUL.md。

26.1 为什么一句“你是一个……”远远不够
这件事很多人一开始不一定会立刻意识到。
因为在非常短、非常单轮的任务里，
一句“你是一个什么什么专家”，常常已经够用了。
比如：


写一个标题


总结一段资料


改一段文案


给一个建议


但一旦你进入的是：


多轮对话


多角色协作


多项目切换


长期运行


持续被调用


问题就出来了。
因为一句“你是一个项目经理”并不能自动回答这些更关键的问题：


你到底只负责哪个层级的事？


你遇到什么情况必须上报？


你能不能替项目拍板？


你有没有权利越过项目主管直接给主人建议？


你是不是可以接触所有资料？


你说话应该偏分析、偏执行、还是偏汇报？


当信息不完整时，你应该推断、提问、还是升级？


你会发现，真正决定 Agent 行为稳定性的，不是标签，
而是：
边界。
而一句“你是一个……”最大的局限就在于：
它给了一个身份标签，
却没有给出足够清楚的边界结构。
所以当系统变复杂时，
Agent 就会开始自己补脑、自己发挥、自己延展。
这就是漂移的起点。

26.2 SOUL.md 到底是什么
为了让这个概念更容易理解，我还是沿用前面第 0 章的“人话翻译”方式。
如果模型是这个 Agent 的大脑，
那 SOUL.md 可以理解成它的：


岗位说明书


行为守则


工作风格说明


升级规则


边界说明


协作约定


也就是说，SOUL.md 不是一个“写点设定词”那么简单的文件。
它本质上是在回答：

这个 Agent 在这家公司里，到底是什么角色？

以及更进一步：

它该怎么工作，
它不该怎么工作，
它什么时候应该停手，
它什么时候应该升级，
它和谁协作，
它最终交什么结果出来？

所以我会说，SOUL.md 真正重要的地方，不是“让 Agent 更有个性”，
而是：
让 Agent 更有边界。
没有 SOUL.md 的 Agent，
很多时候就像一个刚入职、岗位模糊、规则模糊、上下级关系也不清楚的员工。
他当然也能干活，
但越干越容易越界。
而 SOUL.md 的作用，就是尽可能减少这种越界和漂移。

26.3 为什么 SOUL.md 比“提示词更长一点”更重要
很多人第一次接触 SOUL.md，会下意识觉得：

这不就是一个写得更长的 prompt 吗？

表面上看好像有点像。
但从组织设计角度，它和普通 prompt 最大的区别在于：
普通 prompt 更像任务指令，
SOUL.md 更像角色制度。
这两者差别很大。
Prompt 更像什么
像你对一个人说：


帮我做这件事


按这个方式输出


注意这几个要求


它更偏任务级。
SOUL.md 更像什么
像你告诉这个人：


你是谁


你归哪层


你负责什么


你不负责什么


你遇到哪些情况必须升级


你和哪些角色协作


你要以什么方式输出结果


它更偏角色级。
任务级 prompt 当然需要。
但如果没有角色级规则，
系统就会很容易变成：


每次都临时解释


每次都靠上下文猜


每次都可能行为不一致


而一旦有了 SOUL.md，
Agent 的很多基础行为会稳定很多。
所以我会建议你把它理解成：
SOUL.md 不是任务提示的补充，
它是角色稳定性的底层文件。

26.4 一个好的 SOUL.md，最少应该回答哪些问题
这里我建议讲得很具体。
因为很多人一说“写角色文件”，就容易写得又大又空。
在我看来，一个真正有用的 SOUL.md，至少要回答下面这些问题。
第一，你是谁
也就是角色定义。
不是写一堆形容词，
而是明确：


你在组织里的位置


你扮演什么角色


你服务谁


你不是谁


比如：


你是项目主管，不是 CEO


你是 Builder，不是架构总控


你是客服代理，不是品牌决策者


第二，你负责什么
也就是职责范围。
要尽量写成：


你持续负责哪类问题


你的核心目标是什么


你的日常动作是什么


第三，你不负责什么
这一条非常重要，而且经常被忽视。
如果你不写“不负责什么”，
Agent 很容易默认自己“也可以顺便处理一下”。
而很多越权，就是从“顺便”开始的。
第四，你的输入是什么
你应该接收什么样的信息、任务、上下文。
不是所有东西都应该塞给所有 Agent。
第五，你的输出是什么
你最终交付的，应该长成什么样。
是：


草稿


摘要


结构化汇报


评审意见


代码修改建议


风险提示


升级请求


输出不清楚，Agent 很容易在结果形式上飘。
第六，什么时候必须升级
这决定它会不会把不该自己扛的事硬扛着，
或者什么都来烦你。
第七，你和谁协作
这能帮助它理解自己在组织里的相邻角色是谁。
第八，有哪些明确禁止事项
比如：


不要替创始人拍板


不要擅自承诺外部结果


不要跨项目调用状态


不要输出未验证结论当成事实


这类禁止事项，往往特别值钱。
你会发现，一个好的 SOUL.md 不是“文风很帅”。
而是：
角色清楚、边界清楚、输入输出清楚、升级条件清楚。

26.5 为什么“不负责什么”通常比“负责什么”更值钱
这一点我特别想单独拎出来。
很多人写角色说明时，会很自然地一直写：


你要负责……


你要擅长……


你要帮助……


你要支持……


这些都没问题。
但如果你只写这些，角色通常会越来越膨胀。
因为从 Agent 的角度看，
一旦没有明确禁止，它很容易沿着“帮助用户更多一点”的方向继续外扩。
于是你会慢慢看到这些现象：


项目主管开始像 CEO


客服开始像销售


研究 Agent 开始像决策者


Builder 开始替你定义产品方向


评审 Agent 开始自己重写全部内容


这时候你表面上感觉：
“它挺积极、挺主动。”
但长期看，这往往是在越界。
所以我越来越觉得，SOUL.md 里最值钱的一部分，通常不是：


你能做什么


而是：


你不能做什么


这就像现实公司里，岗位边界有时比岗位能力更重要。
因为能力越强的人，一旦边界不清，越容易无意间把组织搞乱。
所以我会建议，任何一个重要 Agent 的 SOUL.md 都至少写一段：
边界 / 不负责事项 / 禁止行为
这会极大提升系统稳定性。

26.6 为什么升级条件一定要写进 SOUL.md
前面第 12 章我们已经讲过升级流。
到了这里，要把它真正落地进角色文件。
因为如果升级规则只存在于你的脑子里，
系统最后还是会出问题。
一个 Agent 再聪明，如果你没有明确告诉它：


什么情况别自己硬扛


什么情况必须交给项目主管


什么情况必须交给 CEO


什么情况必须交给主人拍板


它就只能靠上下文猜。
而升级这种事，最怕猜。
因为猜轻了，会压着不报。
猜重了，会什么都来烦你。
所以我会建议你把升级条件直接写进 SOUL.md，
而且写得尽量具体。
比如：


涉及预算时必须升级


涉及跨项目资源冲突时必须升级


涉及高风险外部承诺时必须升级


无法判断当前任务是否偏离项目目标时必须升级


信息不足以形成可靠判断时，不能假装确定，必须升级或请求补充


这样做的好处非常大。
因为这会让升级流从“组织理念”变成“角色行为规则”。
一旦写进去，系统的稳定性会明显提高。

26.7 为什么 SOUL.md 不能写得太空，也不能写得太满
这里是一个特别现实的写作平衡问题。
有些人写 SOUL.md，写得很空。
比如：


你是一个专业的项目经理


你负责帮助项目顺利推进


你要保持高效、专业、严谨


这些句子听起来都对，
但真正执行时几乎没什么约束力。
另一些人会走到另一个极端，
把 SOUL.md 写成：


超长规则清单


什么情况都试图穷尽


细到每个动作都写死


写得像一份极其僵硬的操作手册


这也有问题。
因为太满之后，角色会变得：


很难维护


很难更新


很难读懂重点


很容易在实际运行中变成假规则


所以一个好的 SOUL.md 要找到一个平衡：
不能太空
否则角色没有真正边界。
不能太满
否则角色没有弹性，也不利于维护。
我更建议你把它写成一种“高信息密度的角色说明书”，
重点放在最关键的东西上：


角色定义


核心职责


核心边界


典型输入


典型输出


升级条件


禁止事项


抓住这些，通常已经很有用。
剩下更细的任务差异，可以交给具体 prompt 或工作流去处理。

26.8 什么样的 Agent 最值得优先写好 SOUL.md
理论上所有核心 Agent 都该有。
但如果你的系统还在早期，不一定一开始就要把几十个角色全部写到同样细致。
我会建议你优先把下面这些写好：
第一类：CEO Agent
因为它决定高层汇总和调度是否稳定。
第二类：项目主管 Agent
因为它是整个项目推进的关键中层。
第三类：共享中心关键角色
比如：


Architect


Builder


QA


Growth Lead


Research Lead


Knowledge Curator


这些角色一旦飘，整个共享体系都会受影响。
第四类：高风险前台角色
比如客服类、对外沟通类、可能涉及承诺和风险的角色。
第五类：评审与融合类角色
因为它们特别容易“越俎代庖”，从评审变成重写，从判断变成拍板。
也就是说，不是所有 SOUL.md 都必须同时做到最精，
但你一定要优先抓住那些：
一旦漂了，就会明显影响系统秩序的角色。

26.9 一个成熟的 SOUL.md，最终应该起到什么作用
如果把这一章收束起来，我会说，一个成熟的 SOUL.md，最终应该起到四个作用。
第一，稳定角色
让这个 Agent 长期看起来像同一个岗位，
而不是今天像这样、明天像那样。
第二，保护边界
避免越权、越层、越项目、越责任。
第三，降低管理解释成本
你不用每次都重新解释：


你是谁


你该做什么


你不该做什么


第四，让系统更容易复制
因为有了角色文件，
你以后要开新项目、加新角色、迁移系统时，会容易很多。
所以 SOUL.md 不是一个“锦上添花的文档”。
它是整个 AI 公司里非常关键的一种基础设施。
它不会直接带来最炫的演示效果，
但它会决定你的 Agent 到底是“像员工”，还是“像一团聪明但飘忽的能力”。
而真正的组织，最怕的往往不是不够聪明，
而是：
边界不稳。
SOUL.md 的意义，就是尽量把这种不稳压下去。

本章小结
SOUL.md 不是一个“更长的提示词”，
而是一个 Agent 的：


岗位说明书


行为守则


边界定义


升级规则


协作约定


一个成熟的 SOUL.md，至少应该写清楚：


你是谁


你负责什么


你不负责什么


你的输入是什么


你的输出是什么


什么时候必须升级


你和谁协作


有哪些禁止事项


其中尤其重要的是：


不负责什么


升级条件


禁止事项


因为这些往往最能防止角色漂移和越权。
系统能不能长期稳定，很多时候不只是看模型，
而是看这些 Agent 有没有被真正写进组织边界里。
接下来，我们就要继续往下，进入另一个很落地的部分：
项目模板、注册表和目录结构。
也就是：
新项目怎么不是每次都从零搭起。
